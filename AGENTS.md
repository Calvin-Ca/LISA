# AGENTS.md

本仓库的协作与执行约束如下，适用于所有编码代理和本地开发流程。

## 执行环境

- 本地 Windows 机器只用于编写代码、修改文档、阅读项目结构和做轻量级静态检查。
- 模型训练、GPU 推理、SAM / GroundingDINO / LISA 权重加载、数据集下载和长时间运行任务全部在远程 Linux + GPU 服务器执行。
- 本地不要运行 `train_ds.py`、`chat.py`、`app.py`，不要加载大模型权重，不要下载大型数据集或模型文件。
- 可以在本地运行不依赖 GPU、权重和大数据的纯逻辑校验，例如数据格式检查、路径检查、`--dry-run` 流程或小型单元测试。

## 开发要求

- 写代码时默认目标运行环境是远程 Linux + GPU，注意路径兼容、CUDA 依赖和大小写敏感文件名。
- 本地提交的变更应保证服务器同步后可直接运行，运行命令默认提供给远程服务器执行。
- 训练相关命令、推理命令和依赖安装命令应明确标注为“远程执行”。
- 不要把服务器私有绝对路径、密钥、令牌、权重文件或数据集大文件写入仓库。
- 编码代理连接远程服务器前必须先获得用户当次确认；服务器地址、账号和认证凭据从仓库外的本机 SSH 配置或安全凭据存储中读取，不写入仓库。
- 给服务器粘贴执行的短命令优先写成单行 `python -c "..."` 或普通 shell 命令，避免使用 heredoc（如 `python - <<'PY' ... PY`），因为缩进或结束标记不顶格会让 shell 卡在 `>` 续行提示。
- 给服务器粘贴执行的长命令需要换行时，必须使用行末反斜杠 `\` 显式续行；`\` 必须是该行最后一个字符，后面不得有空格或注释。
- 不得在命令参数、文件名或路径中间直接换行；换行应放在参数边界，并保证每个路径在同一行内完整呈现。

## 实验执行闭环

- 执行任何训练、推理或评估实验前，先整理实验背景、实验配置、数据划分、模型/权重来源、输出目录、预期产物和完整执行脚本。
- 在用户确认前，不直接修改对应实验目录的 `EXPERIMENT.md` 和 `command.sh`，也不要求用户在服务器上执行实验。
- 用户确认后，再把实验背景和配置写入 `EXPERIMENT.md` 的对应位置，把完整执行脚本写入 `command.sh`。
- 用户确认实验方案后，代理可直接将相关实验文件提交并推送到远端仓库，不需要再次询问是否提交或推送。
- 任何实验执行脚本必须自包含：脚本内部要显式写出所需变量、路径和参数，不依赖用户提前在 shell 中 `export` 环境变量。
- 自包含脚本仍不得把服务器私有绝对路径、密钥、令牌、权重文件或数据集大文件写入仓库；优先使用仓库相对路径，必要时使用用户确认过的可迁移占位路径。
- 用户在远程服务器执行实验后，将结果或日志反馈给代理；代理再根据 `outputs/summary.json`、日志和 bad case 分析完善 `EXPERIMENT.md` 的核心指标、结论和备注。

### 单样本审核输入

每个 Rank 必须同时查看以下内容，不能只根据文件名或原 Prompt 改写：

1. `Sample`：确认图片中实际存在的人、设备、构筑物、空间位置和实例数量。
2. `JSON`：查看当前 `text`、`shapes` 多边形及 `source.sample_key`。
3. `Visualization`：对照 GT mask 和模型预测，确认多边形实际覆盖的对象；Visualization 仅用于理解和发现 bad case，不能反向决定真实标签。
4. 必要时对原图叠加 JSON polygon 做本地轻量可视化；不得加载 SAM、LISA 或其他大模型权重。

### Prompt 标注规则

- Prompt 必须准确描述 mask 实际覆盖的对象，而不是笼统复述 COCO 类别。mask 标人就写“人员”，标洞口、楼板边缘或设备就写对应对象；只有 mask 确实覆盖一片空间时才写“区域”。
- 明确实例数量：一个、两个或多处目标不得在不同 Prompt 中互相矛盾；多实例 mask 不得只描述其中一个实例。
- 方位、颜色、设备关系和场景细节只在图中清晰可见、能够消除歧义且与 mask 一致时使用。不得推断图片无法证实的动作、因果、违规状态或风险关系。
- 同一样本的所有 Prompt 必须严格等价，指向相同像素、相同实例和相同目标粒度。多样性只来自句式和措辞，不得改变分割任务。
- 默认写入 6 条 Prompt：3 条视觉定位型、2 条安全风险型和 1 条上游 Agent 自然查询型。若图像或 mask 存在歧义，宁可只保留少量可靠描述并将该 Rank 标记为待确认，不为凑数量而猜测。
- 原 Prompt 若准确可以改写后保留；若过于抽象、对象错误或与 mask 不一致，应从 `text` 中移除，不能用 JSON 注释停用。
- JSON 中禁止 `//` 或 `/* ... */` 注释。所有 Prompt 使用双引号字符串，条目之间使用英文逗号。

### Prompt 标注完整方法

#### 1. 建立证据链

按以下顺序获取并交叉核对证据：

1. 从 `samples_by_iou.md` 根据 Rank 定位 Sample、JSON、Visualization、类别和评估指标。
2. 查看原图，列出可见候选目标、方位、数量、颜色、结构和相互关系；此时不直接写 Prompt。
3. 查看 JSON polygon/mask，确认真正被覆盖的像素、对象、实例数量和目标粒度。原图说明“画面里有什么”，mask 决定“训练要求分割什么”。
4. 使用 `source.sample_key` 和 `source.source_category` 确认安全类别，它们用于补充风险语义，不能覆盖或篡改 mask 的视觉事实。
5. 查看 GT/预测对比图，用于发现原 Prompt 造成的歧义、误检和漏检；模型预测不得反向决定真实标签。

证据冲突时不得强行生成 Prompt：原图、类别和 mask 无法形成唯一解释的样本，进入待确认状态。

#### 2. 先定义唯一分割目标

生成任何句子前，先在工作记录中明确以下五项：

- `target_object`：mask 实际覆盖的具体对象，例如人员、洞口挡边、楼板边缘、设备或材料堆。
- `instance_count`：单个、两个或多处目标。
- `visual_anchor`：能够消除歧义的方位、颜色、结构或邻近对象。
- `mask_granularity`：人员整体、设备整体、构件、边沿还是一片空间区域。
- `risk_semantics`：由类别和可见事实共同支持的安全含义；无法可靠判断时留空。

这五项构成该样本的语义不变量。6 条 Prompt 可以改变句式，但不得改变这些不变量。

#### 3. 按 3+2+1 组成 Prompt 集

默认 `text` 包含 6 条：

1. 视觉定位型 1：标准描述，直接写清方位、对象和结构。
2. 视觉定位型 2：使用等价句式和近义词，保持相同对象粒度。
3. 视觉定位型 3：使用另一组可靠视觉锚点消除实例歧义。
4. 安全风险型 1：在具体对象前后加入类别支持的风险或防护缺失语义。
5. 安全风险型 2：使用另一种业务风险表达，但仍明确要求分割具体对象。
6. Agent 查询型 1：模拟上游巡检 Agent 的自然查询，例如“定位需要补充洞口防护的位置”，同时保留足以唯一对应 mask 的对象描述。

安全语义 Prompt 推荐结构：

```text
安全状态或风险 + 具体分割对象 + 可靠视觉锚点
```

不得只写“危险区域”“不安全目标”“违规位置”等抽象词。安全语义只能作为具体对象的限定或业务解释，不能取代分割对象。

#### 4. 类别与风险语义边界

- `opening_unprotected`：可使用未封闭、未加盖、洞口防护缺失、人员坠落风险；不得虚构人员正在坠落。
- `guardrail_missing`：可使用临边防护缺失、护栏未设置、坠落风险；必须明确 mask 标的是边沿、开口还是对应构件。
- `harness_missing`：可使用高处作业防坠措施不足、未使用安全带；只有图中确有对应人员且 mask 标人时才这样描述。
- `helmet_missing` / `no helmet`：可使用头部防护缺失、未佩戴安全帽、头部伤害风险；目标必须是 mask 中的人员。
- `no jacket`：可使用未穿反光背心、可视性防护不足；不得写成未穿普通外套。
- `equipment_proximity`：可使用人机距离过近、碰撞或挤压风险；必须由图中人员、设备关系和 mask 共同支持。
- `poor_housekeeping`：可使用材料堆放混乱、通道受阻、绊倒风险；不得把正常材料或整片地面泛化为隐患。
- `safe`：只能描述图中和 mask 支持的合规对象或安全状态，不得为了风险多样性加入不存在的隐患。
- `unsafe`：必须先根据原图和 mask 具体化不安全对象；无法具体化时标记 `instance_ambiguous` 或 `prompt_mask_mismatch`，不得只批量扩写“危险区域”。

#### 5. 反向一致性检查

逐条 Prompt 独立回答以下问题，任一项为“否”就改写或删除：

1. 交给不了解上下文的标注员，是否会选择与当前 mask 相同的对象？
2. 是否保持相同实例数量、空间范围和对象粒度？
3. 是否可能误指向图片中的显著干扰物、整片地面或背景？
4. 风险描述是否由类别和可见事实支持，而不是常识性臆测？
5. 去掉风险修饰后，句子中是否仍有明确、可分割的具体对象？
6. 6 条 Prompt 是否真正具有句式或业务表达差异，同时保持严格语义等价？

#### 6. 写回与校验

- 只把通过检查的 Prompt 写入 JSON `text` 数组；不保留错误 Prompt，不用注释停用旧 Prompt。
- 每个样本修改后比较 Git 版本，确认变化字段只有 `text`；`shapes`、`source`、`is_sentence` 等必须保持不变。
- 校验 JSON 可解析、`text` 非空、条目均为非空字符串、条目不重复，并执行 `git diff --check`。
- 当前训练默认从 `text` 中随机抽取最多 `num_classes_per_sample` 条，长期可覆盖视觉型和风险型表达，但单次采样不保证两类都出现。若未来要求每次强制包含风险型 Prompt，需要增加 Prompt 类型元数据并改造 sampler，不能依赖数组顺序实现。

#### 7. 批量处理原则

- 每批仍按连续 Rank 处理，不能只挑容易样本。
- 先逐样本完成证据链和目标定义，再生成 Prompt；禁止仅按类别对整批做无图改写。
- 同一类别可以复用风险词表和句式框架，但视觉锚点、实例数量和 mask 粒度必须逐图确认。
- 批次报告除 bad case 状态外，还需统计视觉定位型、安全风险型和 Agent 查询型 Prompt 数量，确认 3+2+1 结构得到执行。

### Bad case 判定与处理

审核时为每个样本在工作报告中选择一个主要状态：

- `prompt_ok`：现有 Prompt 与 mask 一致，无需修改。
- `prompt_rewritten`：目标明确，仅需把 Prompt 具体化或补充等价表达。
- `prompt_mask_mismatch`：Prompt 指向对象与 mask 覆盖对象不同，停止修改并请求人工判断。
- `mask_overflow`：mask 明显越界或包含无关对象。
- `mask_missing`：mask 漏掉 Prompt 所指的明显目标或实例。
- `instance_ambiguous`：多实例选择标准无法从现有数据确定。
- `target_unrecognizable`：分辨率、遮挡或画面内容不足以可靠判断目标。

除 `prompt_rewritten` 外，可能涉及标签真值判断的状态不得自行修改 `shapes`；先保留现场并在批次汇报中列出问题。

### 批次执行与验收

1. 每批默认审核 10 个连续 Rank，首批为 51～60。
2. 修改前记录 Rank、JSON 文件名、原 `text` 和主要状态；只对 `prompt_rewritten` 样本写入经过核对的新 `text`。
3. 修改后运行纯逻辑校验：全部目标 JSON 可解析、`text` 为非空字符串数组、Prompt 不重复、禁止注释、除 `text` 外字段与修改前一致。
4. 批次完成后汇报：已审核范围、实际修改 Rank、无需修改 Rank、待用户确认 Rank、各 bad case 数量，以及下一起始 Rank。
5. 在用户确认批次结果前，不自动继续下一批，不执行训练、推理或评估，也不自动提交和推送。
